# 0001：采用混合式环境配置学习法

## 状态

已采纳。

## 日期

2026-06-18

## 背景

本项目是一个多智能体算法竞技场，涉及后端、前端、数据库、缓存、代码执行沙箱、LLM Provider、实时通信等多个技术域。

如果环境配置完全由 AI 助手代劳，短期推进速度较快，但开发者容易错过关键学习点，例如：

- Python 虚拟环境如何工作；
- `.env.example` 和 `.env` 的区别；
- FastAPI 服务如何启动；
- Docker Compose 如何连接 PostgreSQL、Redis、Judge0；
- API Key 和敏感信息应该如何管理；
- 配置失败时如何定位问题。

如果环境配置完全由开发者独立摸索，则容易在工具链和外部依赖上消耗过多时间，影响项目持续推进。

## 决策

采用 **方案 C：手动配置优先的混合式环境配置学习法**。

具体分工：

| 事项         | 负责人          | 说明                               |
| ------------ | --------------- | ---------------------------------- |
| 配置结构设计 | AI 助手主导     | 包括目录、模板、命令说明、排错路径 |
| 敏感信息填写 | 开发者负责      | 包括 API Key、真实密码、私有 Token |
| 关键命令执行 | 开发者手动执行  | 目的是建立真实操作经验             |
| 错误排查     | AI 助手带着排查 | 解释错误含义，而不是只给修复结果   |
| 文档和复盘   | 双方协作        | 每轮配置都记录到开发日志           |

## 原则

1. 先跑通最小闭环，再接复杂外部依赖。
2. 每个配置步骤都要说明命令作用和验证方式。
3. 不提交真实 `.env`、API Key、数据库密码等敏感信息。
4. 配置错误优先作为学习材料，而不是单纯绕过。
5. 每轮环境配置后更新 `docs/开发日志.md`。
6. 除非开发者明确要求，AI 助手不自动代跑配置、安装、启动和测试命令。
7. AI 助手应先给出命令、解释命令、说明预期结果，再由开发者手动执行并反馈结果。

## 当前落地方式

第一阶段只关注后端本地最小环境：

- Python 3.11+
- 项目根目录 `.venv`
- `backend/requirements.txt`
- `backend/.env.example` → `backend/.env`
- `pytest`
- `uvicorn app.main:app --reload`

暂不强制接入：

- PostgreSQL
- Redis
- Judge0
- 真实 LLM API

这些外部依赖会在对应代码抽象稳定后逐步接入。

## 影响

正面影响：

- 项目推进不会被环境问题长期阻塞。
- 开发者能逐步掌握真实工程配置能力。
- 文档会记录每次配置背后的原因和排错过程。

代价：

- 初期会比“完全代劳”稍慢。
- 每次配置后需要额外写少量复盘。

## 后续行动

1. 按 `docs/环境配置学习指南.md` 配置后端本地环境。
2. 将实际执行结果记录到 `docs/开发日志.md`。
3. 后续接入 PostgreSQL、Redis、Judge0、LLM API 时继续新增决策记录或学习笔记。
4. 切换对话后优先读取 `docs/学习协作流程.md` 恢复协作方式。
