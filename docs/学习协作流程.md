# 学习协作流程

本文档是本项目“边构建边学习”的统一入口。以后即使切换对话，也优先按本文档继续推进。

## 一、核心原则

本项目采用 **你手动配置和验证，我指导、解释、排查、整理文档** 的学习方式。

目标不是让 AI 一次性代劳，而是让你在真实项目推进中逐步建立工程直觉：

- 命令为什么这样执行；
- 配置文件为什么这样写；
- 测试为什么这样设计；
- 报错应该从哪里定位；
- 每个切片学到了什么。

---

## 二、角色分工

| 事项     | 你负责                               | 我负责                         |
| -------- | ------------------------------------ | ------------------------------ |
| 环境配置 | 亲自执行命令、填写本机配置、贴出结果 | 给出步骤、解释命令、分析错误   |
| 代码实现 | 可以选择自己写或让我写               | 讲清设计、必要时实现最小切片   |
| 测试验证 | 优先由你手动运行命令                 | 告诉你运行什么、怎么看结果     |
| 错误排查 | 提供完整报错和执行上下文             | 定位原因、解释机制、给修复路径 |
| 文档复盘 | 补充自己的理解和问题                 | 整理流程、决策、学习笔记       |

默认规则：

> 除非你明确要求我执行配置命令，否则环境配置、依赖安装、服务启动、测试命令优先由你手动执行。

---

## 三、每个学习切片的固定流程

每个切片都按以下步骤推进：

```text
1. 明确目标
2. 先讲概念
3. 设计最小实现
4. 编写或修改代码
5. 你手动运行验证命令
6. 根据结果排查问题
7. 更新开发日志 / 学习笔记
8. 进入下一个切片
```

### 1. 明确目标

每个切片开始前先说明：

- 学习目标：这一步要理解什么；
- 构建目标：这一步要产出什么；
- 验证方式：怎么证明它工作；
- 暂不处理：哪些复杂内容先不碰。

### 2. 先讲概念

开始写代码前，先解释相关概念。

例如数据库切片要先讲：

- SQLAlchemy `Base` 是什么；
- `engine` 是什么；
- `session` 是什么；
- 模型类和数据库表是什么关系。

### 3. 设计最小实现

每次只做最小可验证版本。

例如：先做 `Problem` 模型，不一口气做完整比赛系统。

### 4. 编写或修改代码

代码可以由我写，也可以由你写。

如果我写，必须解释关键代码逻辑。

如果你写，我负责 review、解释和排错。

### 5. 你手动运行验证命令

后续默认由你运行命令，例如：

```bash
python -m pytest backend
```

或在本项目当前根目录虚拟环境下：

```bat
.venv\Scripts\python.exe -m pytest backend
```

你把结果贴回来，我再解释输出。

### 6. 根据结果排查问题

排查时优先保留完整上下文：

- 执行目录；
- 执行命令；
- 完整报错；
- 最近修改过的文件。

### 7. 更新文档

每个切片结束后至少更新一个地方：

- `docs/开发日志.md`：记录今天做了什么、命令结果、问题和学习点；
- `docs/学习笔记/`：沉淀可复用概念；
- `docs/决策记录/`：记录重要架构或流程决策。

## 四、当前项目约定

### 虚拟环境位置

当前实际虚拟环境位于项目根目录：

```text
.venv/
```

不是：

```text
backend/.venv/
```

你的 Git Bash 会通过 `.bashrc` 自动激活该环境，但 Cline 工具命令运行在新的 Windows CMD 会话中，不会继承 Git Bash 的自动激活状态。

因此如果需要明确指定环境，使用：

```bat
.venv\Scripts\python.exe -m pytest backend
```

### Python 版本

当前检测到测试运行在 Python `3.14.4`。

测试可以通过，但依赖会产生一些 `DeprecationWarning`。后续可以评估是否切换到 Python `3.11` 或 `3.12` 作为长期开发版本。

### 当前测试基线

最近一次测试结果：

```text
5 passed, 11 warnings in 1.97s
```

包含：

- `backend/tests/test_config.py`：3 个测试；
- `backend/tests/test_health.py`：2 个测试。

## 五、后续推荐切片顺序

当前建议继续从后端基础能力开始：

1. 数据库基础切片：理解 SQLAlchemy async、`Base`、`engine`、`session`。
2. 最小数据模型切片：实现 `Problem` 模型及模型测试。
3. API 路由骨架切片：实现基础 `APIRouter` 和健康/题目路由结构。
4. 数据库初始化 / 迁移切片：再引入 Alembic。
5. Agent 抽象接口切片：实现 `BaseAgent` 和 MockAgent。

推进原则：

> 先用 SQLite 和 Mock 跑通主链路，再接 PostgreSQL、Redis、Judge0 和真实 LLM。

新对话开始时，优先让 AI 读取：

```text
docs/学习协作流程.md
docs/开发日志.md
docs/实施计划.md
```

然后确认：

```text
1. 上一个切片完成到哪里？
2. 最近一次测试结果是什么？
3. 当前下一步是什么？
4. 今天是要学习、实现、排查，还是整理文档？
```
